이번 배포 경험에서 가장 좋았던 점은 GitHub와 연결된 프로젝트가 얼마나 빠르게 인터넷에 올라가는지 직접 확인할 수 있었다는 것입니다. 코드를 푸시하고 잠시 기다리니 실제로 접근 가능한 블로그 주소가 생겼고, 배포가 더 이상 막연한 개념이 아니라는 느낌을 받았습니다.

처음에는 Vercel 로그인 과정에서 Social Account 관련 메시지가 나와 조금 헷갈렸지만, GitHub 계정으로 새 Vercel 계정을 만드는 흐름이라는 것을 이해하고 나니 해결할 수 있었습니다. 그다음에는 GitHub 저장소를 가져와 배포하는 과정이 예상보다 단순해서 초보자에게 좋은 시작점이라는 생각이 들었습니다.

## 인상 깊었던 장면

배포가 완료된 뒤 Vercel 대시보드에서 `Production Deployment` 상태가 초록색으로 표시되고, 배포 주소가 링크로 보이는 화면을 보았을 때 작은 프로젝트라도 실제 서비스처럼 느껴졌습니다. 로컬에서만 보던 블로그가 브라우저 주소창에 공개 URL을 가지고 있다는 사실이 꽤 인상적이었습니다.

> 이번 첫 배포는 "화면을 띄웠다"는 의미를 넘어서, GitHub와 Vercel이 연결된 실제 배포 파이프라인을 처음 경험했다는 점에서 의미가 컸습니다.

## 배포 후 바로 한 일

배포가 끝난 뒤에는 단순히 화면이 보이는 것에서 끝내지 않고, 실제 배포 URL을 프로젝트 코드에도 반영했습니다. `metadataBase`, `sitemap`, `robots` 설정을 실제 주소에 맞추면서 검색 엔진과 배포 환경을 함께 생각해야 한다는 점을 배웠고, 앞으로는 이 경험을 바탕으로 Ubuntu 서버 배포와 AWS까지 이어서 연습해 보고 싶습니다.

## 실제로 사용한 명령

처음 GitHub 저장소를 연결하고 메인 브랜치를 올릴 때는 아래 흐름을 사용했습니다.

```bash
git remote add origin https://github.com/hoyoung94/blog_lab.git
git push -u origin main
```

그다음 실제 Vercel 주소를 코드에 반영하고 다시 배포할 때는 아래처럼 커밋과 푸시를 진행했습니다.

```bash
git add src/app/layout.tsx src/app/sitemap.ts src/app/robots.ts
git commit -m "Vercel 배포 URL 반영"
git push
```

## 배포 후 수정한 코드

특히 아래 두 부분은 "사이트가 열리는가"와는 별개로, 배포 환경을 더 완성도 있게 만들기 위해 중요했습니다.

```ts
export const metadata = {
  metadataBase: new URL("https://blog-lab-two.vercel.app"),
};
```

```ts
const baseUrl = "https://blog-lab-two.vercel.app";
```

이렇게 맞춰 두면 `sitemap.xml`, `robots.txt`, 그리고 메타데이터가 실제 프로덕션 주소를 기준으로 동작합니다.

## 배운 점 세 가지

1. GitHub와 Vercel을 연결하면 배포 과정이 훨씬 단순해진다는 점을 배웠습니다. 직접 서버에 올리지 않아도 저장소를 기준으로 배포가 자동으로 이어지기 때문에, 초보자가 배포 흐름을 처음 익히기에 부담이 적었습니다.
2. 배포 성공은 화면이 열린다는 것만으로 끝나지 않는다는 점을 배웠습니다. 실제 주소를 `metadataBase`, `sitemap`, `robots`에 반영해야 검색 엔진과 배포 환경이 자연스럽게 연결되고, 프로젝트가 더 완성도 있게 보입니다.
3. 작은 오류와 헷갈림도 배포 경험의 중요한 일부라는 점을 배웠습니다. 로그인 과정의 혼란, Git 설정, 실제 URL 반영 같은 문제를 하나씩 해결하면서 배포는 특별한 사람만 하는 작업이 아니라 차근차근 익힐 수 있는 기술이라는 자신감이 생겼습니다.
