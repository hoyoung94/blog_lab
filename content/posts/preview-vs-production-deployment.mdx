Vercel 대시보드를 보다 보면 `Preview Deployment`와 `Production Deployment`라는 표현을 자주 보게 됩니다. 처음에는 둘 다 "배포"처럼 보여서 비슷하게 느껴지지만, 실제로는 용도와 쓰는 시점이 조금 다릅니다.

## 한 줄로 정리하면

- `Production Deployment`는 실제 서비스에 공개되는 배포입니다.
- `Preview Deployment`는 변경 사항을 미리 확인하기 위한 임시 배포입니다.

즉, 프로덕션은 "지금 사용자들이 보는 버전"이고, 프리뷰는 "배포 전에 확인해 보는 버전"이라고 생각하면 이해하기 쉽습니다.

## Production Deployment는 언제 만들어질까

보통 `main` 브랜치 같은 기준 브랜치에 코드를 푸시하면 Production Deployment가 만들어집니다. 이번 프로젝트에서도 `main` 브랜치에 푸시한 뒤 실제 주소인 `https://blog-lab-two.vercel.app`가 업데이트되었습니다.

Production Deployment는 실제 블로그 방문자가 보게 되는 버전이기 때문에, 문구 수정이나 URL 변경처럼 사소해 보이는 수정도 신중하게 확인하는 습관이 중요합니다.

## Preview Deployment는 언제 유용할까

Preview Deployment는 새 기능을 먼저 확인하고 싶을 때 유용합니다. 예를 들어 글 스타일을 크게 바꾸거나, 이미지 레이아웃을 수정하거나, 새로운 페이지를 추가할 때 바로 프로덕션에 올리기보다 미리보기 주소에서 먼저 확인할 수 있습니다.

특히 협업에서는 더 중요합니다. 다른 사람에게 "이 주소에서 먼저 확인해 주세요"라고 보여줄 수 있기 때문입니다.

## 내가 이해한 차이점 세 가지

1. Production은 실제 사용자용이고, Preview는 검토용입니다.
2. Production 주소는 보통 고정된 대표 주소이고, Preview는 변경마다 새로 생길 수 있습니다.
3. Production은 안정성이 중요하고, Preview는 실험과 확인이 더 중요합니다.

## 이런 흐름으로 생각하면 쉽다

```bash
# 1. 기능 작업
git checkout -b feature/preview-test

# 2. 변경 후 푸시
git push origin feature/preview-test

# 3. Preview Deployment에서 먼저 확인
# 4. 이상 없으면 main에 반영
git checkout main
git merge feature/preview-test
git push origin main
```

지금은 혼자 작업하고 있어도 이 흐름을 익혀 두면 나중에 GitHub 협업이나 팀 프로젝트에서 훨씬 도움이 됩니다.

## 초보자에게 추천하는 사용법

- 홈 화면 문구나 스타일을 크게 바꿀 때는 Preview에서 먼저 확인하기
- 배포 URL, 메타데이터, `robots.txt`, `sitemap.xml`처럼 운영에 영향을 주는 설정은 Production에서 다시 확인하기
- 새 글을 추가할 때도 모바일 화면과 데스크톱 화면을 Preview에서 먼저 보는 습관 들이기

처음에는 둘의 차이가 추상적으로 느껴질 수 있지만, 실제로 몇 번 배포를 해 보면 "Preview에서 확인하고 Production에 반영한다"는 흐름이 자연스럽게 익숙해집니다. 이 구분을 이해하는 것만으로도 배포를 조금 더 안전하게 다루는 감각이 생긴다고 느꼈습니다.
